# Plan 03-01: Fix Product Data Sources — Live WC IDs Throughout

## Context

The server at `server/routes/products.cjs` already uses `per_page: 100` and maps WC products to a shaped format with `id: String(wc.id)` (numeric string IDs like "547").

`ProductContext` fetches from `/api/products` and applies `mapWooProductToAppProduct` on the result. However, `mapWooProductToAppProduct` was written expecting raw WooCommerce API data, but now receives already-shaped server data.

Two components still import from seed data (`data/products.ts`) instead of using live products:
- `components/ShopPage.tsx` — imports `products` from `../data/products` directly
- `components/Navbar.tsx` — imports `products` from `../data/products` for mega menu product selection

This means any product added to cart via ShopPage or Navbar mega menu has a seed data ID like `'bait-salted-clams'` instead of a numeric WC ID like `"547"`, causing `product_id: 0` in WooCommerce orders.

## Tasks

### Task 1: Fix productMapper to handle server-shaped data

The server returns products already shaped (no raw `categories`, `images` arrays in WC format). `mapWooProductToAppProduct` in `utils/productMapper.ts` receives server-shaped data.

The current `mapWooProductToAppProduct` does `id: String(wpProduct.id)` which works. But `mapCategory(wpProduct.categories)` will receive undefined since the server data has `categoryId` not `categories`.

**The ProductContext is the wrong place for this mapper.** The server already shapes products — the client doesn't need to remap. Update `ProductContext` to use server data directly WITHOUT calling `mapWooProductToAppProduct`.

In `context/ProductContext.tsx`:
- Remove the `mapWooProductToAppProduct` import
- Remove the `.map(mapWooProductToAppProduct)` call
- Store server products directly: `setProducts(wooProducts)` (the server already returns Product-shaped data)

The server's `shapeProduct` in `products.cjs` already returns all fields needed by the `Product` interface. No client-side remapping needed.

### Task 2: Fix ShopPage to use live products

In `components/ShopPage.tsx`:
- Remove import of `products` from `../data/products`
- Add `import { useProducts } from '../context/ProductContext';`
- Inside the component, call `const { products, loading } = useProducts();`
- The rest of the component logic stays the same (it filters/displays products)

### Task 3: Fix Navbar mega menu to use live products

In `components/Navbar.tsx`:
- Remove import of `products` from `../data/products` (the Product type import can stay if used, but check if it's still needed)
- Add `import { useProducts } from '../context/ProductContext';`
- Inside the component, call `const { products } = useProducts();`
- The existing mega menu click handler at line 281 `products.find(p => p.id === 'rod-surf-11')` needs updating: since live WC products don't have slug IDs, find by name instead:
  ```tsx
  const product = products.find(p => p.name.toLowerCase().includes('surf') && p.name.toLowerCase().includes('11'));
  ```
  Or more robustly, find by category and pick the first rod:
  ```tsx
  const product = products.find(p => p.categoryId === 'rods' || p.categoryId === 'rod');
  ```
  Use the most specific match available. If no match, skip the navigation.

Note: `Product` type is imported from `../data/products` — this can change to `import { Product } from '../types'` since Product is defined in `types.ts`.
